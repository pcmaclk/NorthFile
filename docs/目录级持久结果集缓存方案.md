# 目录级持久结果集缓存方案

## 目标

- 把当前“目录热快照只在会话内越用越快”的能力，推进到“跨会话复用的目录结果集索引”。
- 优先解决大目录首屏冷加载仍要整目录 `read_dir + sort` 的问题。
- 保持当前 WinUI 侧已经收口好的 `ResultSet + ReadRange(startIndex, count)` 模型，不把 UI 再绑回目录分页细节。

## 当前基础

- 当前 Rust 已具备：
  - 目录按 `path + sort_mode` 的内存热快照
  - 排序后 `Vec` 按索引切片
  - `size / modified` 直接随结果返回
- 当前 WinUI 已具备：
  - 详情模式按视口范围读取
  - `IEntryResultSet`
  - `DirectoryEntryResultSet / SearchEntryResultSet`

也就是说，持久缓存不需要重新发明结果集协议，只需要把当前目录结果集的底层存储从“纯会话内存”向“可复用索引”扩展。

## 设计原则

- 先做目录级，不直接做全局 Everything。
- 先做可失效、可回退、可观测，不追求第一版就全量实时更新。
- 先支持最常用排序：
  - 名称
  - 修改时间
  - 大小
- 缓存命中时优先服务首屏和视口读取，不要求一次性恢复全部目录。

## 结果结构

### Key

- `volume_id`
- `normalized_directory_path`
- `sort_mode`
- `sort_direction`

### Value

- `total_count`
- `entries[]`
  - `name`
  - `is_directory`
  - `is_link`
  - `size_bytes`
  - `modified_unix_ms`
  - 预留：
    - `file_id`
    - `parent_id`
    - `extension_hash`
- `directory_signature`
  - `directory_last_write`
  - `entry_count_hint`
  - 后续可扩展 `usn` / `file_id`
- `created_at`
- `last_access_at`
- `schema_version`

## 第一阶段实现

### Phase 1: 持久结果集缓存壳

- 新增 `PersistentDirectoryResultCache`
- 存储位置建议：
  - `AppData/Local/NorthFile/cache/resultsets/`
- 存储格式先用：
  - 单目录单文件
  - 二进制或紧凑 JSON 都可
  - 第一版优先实现成本，不先追求极致压缩

### Phase 2: 命中判定

- 进入目录时先查：
  - `path + sort_mode`
- 再校验目录签名：
  - 目录最后写入时间
  - 条目数 hint
- 命中则：
  - 直接恢复 `total_count`
  - 直接支持 `read_range(start, count)`
- 未命中则：
  - 走当前内存快照构建
  - 构建完成后异步写回持久缓存

### Phase 3: 淘汰与容量控制

- 先按目录文件数控制：
  - 例如最多保留 `100~300` 个目录结果集
- 淘汰依据：
  - `last_access_at`
- 容量上限：
  - 先做软上限，超出时后台清理

## 与当前内存缓存的关系

- 不替换当前内存热目录缓存。
- 推荐结构：
  - L0: 当前请求块
  - L1: 会话内目录热快照
  - L2: 持久目录结果集缓存

读取顺序：

1. 先查内存热快照
2. 再查持久目录结果集缓存
3. 最后回退到当前冷构建路径

这样不会影响已经很快的热路径，只打首屏冷加载。

## WinUI 集成点

- `IEntryResultSet` 不改接口。
- `DirectoryEntryResultSet` 不感知底层是否来自：
  - 内存热快照
  - 持久目录结果集缓存
  - 冷构建
- WinUI 只继续消费：
  - `TotalCount`
  - `TryReadRange(startIndex, count)`

## 失效策略

第一版只做弱一致性：

- 目录签名不一致则整目录缓存失效
- 命中失败直接回退冷构建

第二版再补：

- 文件系统 watcher
- USN 驱动的增量失效
- 分块重建而非整目录重建

## 验收

- 同一目录第二次进入首屏明显快于第一次
- 应用重启后再次进入热目录，首屏仍快于冷启动
- 排序切换只影响对应排序视图，不污染其他排序结果
- 缓存失效后能平滑回退到冷构建
- `System32` 一类 4000+ 项目录要纳入基准

## 开发顺序建议

1. Rust 持久结果集缓存壳
2. 目录签名与命中判定
3. 写回与淘汰
4. `ExplorerService` / `DirectoryEntryResultSet` 接入
5. perf 日志补充：
   - `persistent-hit`
   - `persistent-miss`
   - `persistent-write`
   - `persistent-evict`
