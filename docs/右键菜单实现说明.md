# 右键菜单实现说明

## 当前结构

- 文件、文件夹、背景三类右键菜单是三套独立的 `CommandMenuFlyout` 实例，不再共用一份菜单后再临时裁剪。
- 顶层右键菜单基于自定义 `CommandMenuFlyout : MenuFlyout`。
- 顶层菜单主体仍然使用原生 `MenuFlyoutItem / MenuFlyoutSubItem`，普通菜单项和子菜单展开尽量保持 WinUI 原生链路。
- 顶层菜单底部的命令按钮栏直接放在 `MenuFlyoutPresenter` 模板里，不额外起 `Popup`。

## 顶层菜单实现

- `CommandMenuFlyout` 定义在 [CommandMenuFlyout.cs](/D:/Develop/Workspace/Rust/FileExplorer/FileExplorerUI/Controls/CommandMenuFlyout.cs)。
- 样式模板定义在 [App.xaml](/D:/Develop/Workspace/Rust/FileExplorer/FileExplorerUI/App.xaml) 的 `CommandMenuFlyoutPresenterStyle` 里。
- 模板里通过两个命名部件承载底部命令栏：
  - `PART_CommandBar`
  - `PART_CommandSeparator`
- `CommandMenuFlyout` 在 `CreatePresenter()` 时拿到这两个模板部件，并把 `Commands` 集合作为按钮栏数据源写进去。
- 子菜单不会走这条注入路径，因此不会显示底部按钮栏。

## 底部命令栏约束

- 所有按钮统一宽度。
- 按钮栏左对齐排列。
- 菜单有最小宽度，按钮总宽度不足时右侧留空。
- 最后一个按钮不画竖分割线。
- 按菜单实际向上/向下展开方向，在顶部或底部切换按钮栏位置。
- `Properties` 统一放在上面的普通菜单区最后一项，不放在底部按钮栏。

## 三类菜单

### 文件

- 顶部普通项：`Open / Open with / Share / Compress / Create shortcut / Copy file path / Set tag / Properties`
- 底部按钮：`Cut / Copy / Rename / Delete`

### 文件夹

- 顶部普通项：`Open / Open in new window / Pin to sidebar / Unpin from sidebar / Compress / Copy folder path / Open in terminal / Set tag / Properties`
- 底部按钮：`Cut / Copy / Paste / Rename / Delete`
- `Paste` 按需显示。

### 背景

- 顶部普通项：`View / Sort by / Group by / New / Open in terminal / Properties`
- 底部按钮：`Paste / Refresh`
- `Paste` 按需显示。

## 当前排序/分组语义

### 类型排序

- `Sort by -> Type` 不是简单按类型列显示文本做字符串比较。
- 当前目标语义是：
  - 文件夹优先
  - 然后按稳定的类型大类排序
  - 同一大类内再按扩展名排序
  - 扩展名相同时再按名称排序
- 类型大类当前按以下顺序排列：
  - 文件夹
  - 图片
  - 视频
  - 音频
  - 文档
  - 压缩包
  - 程序
  - 代码
  - 磁盘映像
  - 其他
  - 未知
- 这样 `png / jpg / webp` 之类会排在同一段里，不再被纯显示文本打散。

### 类型分组

- `Group by -> Type` 和 `Sort by -> Type` 是两套规则。
- 分组决定组间固定顺序和组标题。
- 排序只决定每个组内的条目顺序。
- 当前类型分组顺序是：
  - 文件夹
  - 文档
  - 图片
  - 音频
  - 视频
  - 压缩包
  - 程序
  - 其他
  - 未知

## Paste 显隐

- 文件夹和背景菜单里的 `Paste` 不是静态常显。
- 应用内复制/剪切时会显示。
- 外部程序把文件或文件夹放进系统剪贴板时也会显示。
- 当前判断走 [FileManagementCoordinator.cs](/D:/Develop/Workspace/Rust/FileExplorer/FileExplorerUI/Services/FileManagementCoordinator.cs) 里的 `HasAvailablePasteItems()`。

## Compact 侧边栏目录菜单

- compact 侧边栏里的“我的电脑”目录弹出菜单，和列表右键菜单不是同一套实现。
- 它仍然使用原生 `MenuFlyout / MenuFlyoutSubItem`。
- 创建菜单时只探测下一层是否存在子目录，以决定是否显示箭头。
- 真正展开某层时，才枚举这一层子目录。
- 这一层逻辑现在避免了把 `async` 枚举包装成 `GetAwaiter().GetResult()` 堵在 UI 线程上。

## 下一阶段命令集成表（2026-03-22）

目标是把“右键 / 快捷键 / toolbar / provider / service / Rust 能力”拆成同一张命令清单，避免 WinUI 先写一套、Rust 再补一套。

### 分组

- `纯 WinUI 可推进`
  - 主要涉及菜单挂线、command/provider/can-execute、现有 `ExplorerService` 或系统壳调用。
- `需要应用状态持久化`
  - 主要涉及本地配置读写、sidebar 刷新、重启恢复。
- `需要 Rust 或底层能力扩展`
  - 主要涉及压缩、批量能力、统一结果结构、错误码或高性能实现。

### 命令矩阵

| 命令 | 目标 | 当前状态 | 依赖层 | 优先级 | 备注 |
| --- | --- | --- | --- | --- | --- |
| `Open in new window` | 文件夹 | 已实现（首版） | 纯 WinUI 可推进 | 高 | 当前由应用内多窗口入口直接创建新 `MainWindow` 并传入初始路径；后续再考虑和双面板/会话状态收口 |
| `Create shortcut` | 文件/文件夹 | 菜单占位 | 纯 WinUI 可推进 | 高 | 先做 `.lnk` 创建或最小系统壳调用即可 |
| `Pin to sidebar / Unpin` | 文件夹 | 菜单占位 | 应用状态持久化 | 中高 | 不必先做设置页，但需要 pinned 列表保存、加载、去重和 sidebar 刷新 |
| `Open with` | 文件 | 已实现（系统对话框） | 纯 WinUI 可推进 | 中高 | 当前调用系统 `SHOpenWithDialog`；后续若要接近资源管理器体验，需要补自定义“最近/推荐程序”列表 |
| `Open target` | 快捷方式 | provider 已有，菜单未接 | 纯 WinUI 可推进 | 中 | 适合顺手补齐，和 `ShortcutFileCommandProvider` 已有能力对齐 |
| `Run as administrator` | 可执行文件 | provider 已有，菜单未接 | 纯 WinUI 可推进 | 中 | 可先走系统壳 `runas`，权限失败走状态提示 |
| `Preview` | 图片/音视频 | provider 已有，菜单未接 | WinUI + 预览宿主 | 中 | 若当前没有统一预览面板，可先延后 |
| `Share` | 文件 | 菜单占位 | WinUI + 系统集成 | 中 | WinUI 3 桌面分享集成比普通壳调用更绕，建议晚于前几项 |
| `Compress -> ZIP` | 文件/文件夹 | 菜单占位 | Rust 或服务层扩展 | 中高 | ZIP 可作为第一批压缩能力；建议返回统一结果结构 |
| `Compress -> 7z` | 文件/文件夹 | 菜单占位 | Rust 或外部工具能力 | 中 | 不建议先于 ZIP 落地 |
| `Extract here` | 压缩包 | provider 已有，菜单未接 | Rust 或服务层扩展 | 中 | 与压缩能力一起设计结果结构更稳 |
| `Extract to folder` | 压缩包 | provider 已有，菜单未接 | Rust 或服务层扩展 | 中 | 同上 |
| `Set tag` | 文件/文件夹 | 菜单占位 | 应用状态或数据模型扩展 | 低 | 先不要在当前阶段抢主线 |

### 建议并行方式

- WinUI 线
  - 只负责菜单挂线、provider 可见性、can-execute、状态提示、执行后 UI 刷新。
  - 不直接扩散底层文件系统逻辑。

- 服务层 / C# 协调线
  - 通过 `ExplorerService` / `FileManagementCoordinator` 暴露稳定入口。
  - 统一返回执行结果、错误码、冲突信息、后续 UI 动作（刷新 / 重选中 / 导航）。

- Rust 线
  - 只负责底层能力、性能、批处理、压缩/解压、统一错误语义。
  - 不感知 WinUI 菜单结构。

### 推荐实施顺序

#### 第一批

- `Open in new window`
- `Create shortcut`
- `Open with`
- `Open target`
- `Run as administrator`

这一批基本都可以先由 WinUI + `ExplorerService` 收口，不必等 Rust 先动。

### 第一批当前进展

- `Open in new window`
  - 已完成首版。
  - 当前实现路径：
    - `App` 维护窗口列表，避免新窗口被回收。
    - `MainWindow` 支持通过构造参数接收初始路径。
    - 文件夹右键命令直接创建新窗口并导航到目标目录。
  - 后续可选增强：
    - 继承当前视图模式、排序、分组、查询状态。
    - 明确多窗口与未来双面板/工作区状态的关系。

- `Open with`
  - 已完成首版。
  - 当前实现路径：
    - 通过系统 `SHOpenWithDialog` 弹出系统级“打开方式”对话框。
    - 目标是先保证功能可用，而不是立刻复刻资源管理器完整体验。
  - 当前已知边界：
    - 目前使用的是系统对话框，不控制系统内建的最近项、推荐项和默认应用呈现。
    - 交互样式取决于 Windows 当前版本和该扩展名的系统状态。
  - 后续自定义版待办：
    - 做应用内 `Open with` 对话框，支持“最近使用”“推荐应用”“浏览更多应用”三段。
    - 从扩展名关联信息、`OpenWithList`、`OpenWithProgids`、用户最近选择中拼装候选列表。
    - 为候选项补应用名、图标、可执行路径、去重和失效清理。
    - 点击候选后按选定程序启动目标文件。
    - 视需要再补“始终使用此应用”的持久化与冲突处理。

- `Create shortcut`
  - 已完成首版。
  - 当前实现路径：
    - 通过 `WScript.Shell` 创建 `.lnk`。
    - 目标支持普通文件与文件夹。
    - 复用 `FileManagementCoordinator` 处理重名、刷新与创建后选中。
  - 当前已知边界：
    - 当前仍是当前目录内创建快捷方式，不含桌面/开始菜单等变体入口。
    - 命名策略当前以本地化“快捷方式/Shortcut”后缀为主，后续如要完全对齐系统命名规则，需要再补语言和冲突策略。

- `Open target`
  - 已完成首版。
  - 当前实现路径：
    - `.lnk` 解析 `TargetPath + Arguments`，避免指向 `explorer.exe` 的快捷方式再次打开系统资源管理器。
    - `.url` 解析 `URL=` 并交给系统打开。
    - 目标是目录时回到应用内导航；目标是普通文件或网页时交给系统打开。
  - 当前已知边界：
    - 对少数 `explorer.exe /select,`、`shell:` 之类变体参数，后续仍可继续补更细的参数归一化。

- `Run as administrator`
  - 已完成首版。
  - 当前实现路径：
    - 通过 `ShellExecuteEx + runas` 拉起提升后的可执行目标。
    - 当前仅针对 provider 已识别的可执行目标开放。
  - 当前已知边界：
    - 目前仍是系统级提升启动，不含应用内进程托管或提升后回传状态同步。

#### 第二批

- `Pin to sidebar / Unpin`

这一批主要是“应用状态持久化”问题，不要求 Rust 先配合，但需要先定 pinned 数据格式和存储位置。

#### 第三批

- `Compress -> ZIP / 7z`
- `Extract here`
- `Extract to folder`
- `Share`

这一批更适合和 Rust / 系统集成一起设计，避免先做一版临时 UI 入口，后面再重写执行链。

## 2026-03-23 补充说明

- 第一批右键菜单当前已经完成并可用的项：
  - `Open in new window`
  - `Open with`
  - `Create shortcut`
  - `Open target`
  - `Run as administrator`
- 本轮实现过程中已经形成固定约束：
  - 不能只补 `MainWindow.xaml.cs` 的命令执行逻辑，还必须同步接上 `MainWindow.xaml` 里的动态 `Tag + Click` 路由。
  - 对“包装/壳文件”类型，右键命令和普通 `Open / Enter / 双击` 必须共用同一条解析链，避免再次出现“右键行为和直接打开不一致”。
  - 后续如果继续加类似格式，应优先走：
    - traits / target resolver
    - `ExplorerService` 的格式解析能力
    - `MainWindow` 统一打开入口
  - 不应再为某一类特殊文件单独复制一套仅右键可用的执行逻辑。
