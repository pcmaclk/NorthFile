## 🛠️ 技术栈全景图

| 维度 | 推荐技术 | 核心作用 |
| --- | --- | --- |
| **表现层 (UI)** | **WinUI 3 + XAML** | 实现 Windows 11 原生的 Mica/Acrylic 效果和流畅动画。 |
| **逻辑层 (Glue)** | **C# 12/13 (.NET 8/9)** | 负责 UI 绑定、异步调度、与 Rust 层的通信。 |
| **性能引擎 (Core)** | **Rust** | 负责文件扫描、MFT 解析、重度 I/O 操作、搜索算法。 |
| **编译技术** | **Native AOT** | 将 C# 编译为原生机器码，实现“秒开”启动速度。 |
| **交互桥梁** | **P/Invoke / C-ABI** | 实现 C# 与 Rust 之间的零成本（或低成本）函数调用。 |

## 🚀 关键技术要点 (Key Pillars)

### 1. 内存与数据：零拷贝 (Zero-copy) 设计

千万不要在 Rust 和 C# 之间频繁传递大型 JSON 或字符串。

* **做法：** 在 Rust 中分配一块连续内存（Buffer），通过指针传给 C#。
* **技术：** 使用 C# 的 `Span<T>` 或 `Memory<T>` 直接指向 Rust 的内存地址。这样 UI 渲染时读取文件名，不需要经过“复制”过程。

### 2. I/O 引擎：基于 MFT 的极速搜索

不要用传统的递归 `Directory.GetFiles()`，那是 Files 慢的主因之一。

* **做法：** Rust 底层直接解析 NTFS 的 **MFT (Master File Table)**。
* **优势：** 可以在毫秒级遍历百万级文件。搜索功能应该像 *Everything* 一样作为底层基础设施，而不是一个插件。

### 3. UI 策略：极致的虚拟化 (Virtualization)

WinUI 的 `ItemsRepeater` 是你的好伙伴。

* **做法：** 即使文件夹有 10 万个文件，UI 层只创建当前屏幕可见的 20 个控件。
* **进阶：** 配合 **异步图标加载**。图标生成必须在 Rust 线程池中排队，渲染时先显示通用图标，等 Rust 计算完缩略图后再通过回传指针异步刷新。

### 4. 架构解耦：中转层防抖 (Debouncing)

文件系统是个“多话”的来源（比如编译代码时产生数千个文件变更）。

* **做法：** 在 Rust 侧建立一个事件缓冲区。
* **逻辑：** 100ms 内的所有文件变更合并成一个“变更包”发给 UI。避免 UI 线程因为频繁刷新而掉帧。

---

## ⚠️ 避坑指南 (Avoid the "Files" Trap)

* **瘦 ViewModel：** 保持 C# 的 ViewModel 极度轻量。它只应该包含索引号和最基本的显示属性，复杂的元数据解析全放在 Rust 侧，随用随取。
* **禁用慢速 API：** 尽量避开 `Shell32.dll` 中的旧 API（如获取右键菜单项），它们同步且阻塞。考虑自己用 Rust 实现一个轻量级的上下文菜单逻辑。
* **Native AOT 约束：** 如果追求 Native AOT，注意不要使用太重的反射（Reflection）库，这会影响编译优化。

## 📈 推荐开发第一步

不要先画 UI。先建立 **"The Bridge"**。

1. 创建一个 **Rust Library** 项目，写一个简单的 `search_files` 函数，返回一个字符串指针。
2. 创建一个 **WinUI 3** 项目，尝试通过 `[LibraryImport]` 调用这个 Rust 函数并显示在界面上。
3. 测试 **Native AOT** 编译，确保这个链路是通畅的。
