# 全局索引数据结构与演进方案

## 目标

- 在当前“目录级持久结果集缓存”之上，演进到可跨目录复用的全局索引。
- 保持 WinUI 侧现有 `IEntryResultSet` / `ReadRange(startIndex, count)` 模型不变。
- 让目录浏览、搜索结果、后续筛选视图逐步共用同一套底层数据源。
- 为后续动态更新打基础，但第一阶段不直接追求完整 Everything 级能力。

## 当前状态

- 当前目录结果集本质上是：
  - `path + sort_mode -> Vec<MemoryDirItem>`
  - 持久化后恢复成排序好的目录级数组
- 当前优点：
  - 单目录范围读取快
  - 热目录复用明显
  - 已能跨会话命中
- 当前限制：
  - 真源和排序视图耦合
  - 不适合高频插入/删除/重命名
  - 不适合做跨目录搜索
  - 不适合多排序共享一套基础数据

## 设计原则

- 不直接把“目录级排好序数组”放大成“全局超大数组”。
- 主记录表、排序索引、搜索索引分层。
- `ResultSet` 面向 UI，索引实现细节只在 Rust 内部。
- 第一阶段优先支持：
  - 本地 NTFS
  - 名称排序
  - 基础搜索
  - 稳定随机访问
- 任何增量维护失败时，都必须允许回退到局部重建。

## 总体结构

全局索引建议拆成三层。

### 1. 主记录表

真源，不按任何排序保存。

- `entry_id`
  - 本地 NTFS 优先复合键：`(volume_id, frn)`
  - 非 NTFS 或拿不到时退回内部索引 id
- `parent_id`
- `volume_id`
- `normalized_path`
- `name`
- `is_dir`
- `is_link`
- `size_bytes`
- `modified_unix_ms`
- 预留：
  - `extension`
  - `attributes`
  - `usn`

建议结构：

- `slotmap/slab + HashMap<entry_id, slot>`

原因：

- 需要稳定 id
- 需要快速按 id 找记录
- 删除后不希望让大块数组整体搬移

### `entry_id` 语义与复用检测

这里必须明确边界。

- NTFS 本地并不是“单个 `FRN` 永久稳定”。
- 第一阶段建议把 `entry_id` 视为：
  - `(volume_id, frn)` 作为主身份
  - 再附带一组轻量校验信息：
    - `parent_id`
    - `name_hash`
    - `last_confirmed_unix_ms`
- 作用：
  - 在同一索引会话内尽量稳定识别记录
  - 当文件删除后 `FRN` 被复用时，能通过附加校验发现异常

第一阶段先明确两个约束：

- `entry_id` 在“同一索引代际内”稳定
- 不承诺所有 fallback id 跨重启稳定

这意味着：

- Phase 1 主表可以先以内存索引为主
- 真正的跨重启稳定 identity 要等后续持久主表和 USN 语义更完整后再增强

### 内存与持久化边界

`slotmap/slab + HashMap` 在这里的定位，第一阶段应明确为：

- 内存期主表实现
- 先不直接承诺 mmap 持久主表
- 先不解决“百万级全局主表跨会话零成本恢复”

原因：

- 当前阶段的目标是把全局索引的数据模型定下来
- 不是一次性把运行态结构和持久化格式绑定死

第一阶段可接受的实现边界：

- 运行态主表在内存
- 目录级持久结果集缓存继续承担跨会话加速
- 全局主表持久化留到后续阶段单独设计

### 2. 排序索引

面向“按顺序取第 N 条”和“范围读取”。

每种排序一个独立有序视图，例如：

- 名称升序
- 修改时间降序
- 大小降序
- 路径升序

建议结构：

- order-statistics tree
  - 节点存 `entry_id`
  - 带子树大小

这样可以支持：

- `get nth`
- `read range(start, count)`
- 插入/删除
- 局部重排

如果第一阶段实现成本太高，也可以先用：

- `Vec<entry_id>` 作为排序视图

但这更适合作为过渡，不是最终结构。

### `Vec<entry_id>` 过渡方案的退出条件

这里不能无限期停留在 `Vec`。

第一阶段允许：

- 目录浏览
- 小中等结果集搜索
- 以 `Vec<entry_id>` 作为有序视图

但需要明确切换条件：

- 当单个结果集进入“大量增量插入/删除导致移位明显”的区间时，`Vec` 就不再适合作为长期结构
- 建议先把以下指标写成门槛：
  - 单结果集规模超过 `50k ~ 100k`
  - 或增量更新后单次排序视图维护明显超过目标延迟

也就是说：

- `Vec` 是 Phase 1 过渡结构
- Phase 3 真正做动态更新时，如果这些阈值已被触发，就应切到 order-statistics tree，而不是继续硬撑

### 3. 搜索索引

排序索引不负责搜索。

搜索索引建议单独维护：

- 前缀索引
- 倒排索引
- 或者更轻量的 token / n-gram 索引

第一阶段可先只做：

- `name` 分词/归一化后的倒排
- 基础前缀匹配

搜索结果本质上还是：

- 一组 `entry_id`
- 再套用排序视图或临时排序结果

## 与 ResultSet 的关系

UI 不直接碰全局索引内部结构。

### 目录浏览

- `DirectoryEntryResultSet`
  - 先可继续走目录级结果集缓存
  - 后续切到“全局索引中的目录视图”

### 搜索结果

- `SearchEntryResultSet`
  - 直接由搜索索引 + 排序视图产出

### UI 契约保持不变

- `TotalCount`
- `TryReadRange(startIndex, count)`

也就是说，全局索引的目标不是改 UI，而是替换 `ResultSet` 背后的数据来源。

## 分阶段演进

### Phase 1: 主表 + 名称排序视图

目标：

- 先把“目录级缓存”和“基础全局主表”接起来
- 只支持最常用的名称排序
- 优先保证目录浏览与基础搜索可共用一套底层主表
- 在这一阶段就把 `entry_id` 语义、FRN 复用检测、fallback id 生命周期说清楚

范围：

- 本地 NTFS
- `entry_id + path + name + is_dir + size + modified`
- 名称排序视图
- `ReadRange` 可直接从排序视图取 `entry_id` 后映射到主记录

### Phase 2: 搜索索引

目标：

- 在不改 UI 的前提下，把搜索结果从“目录扫描”推进到“全局索引查询”

范围：

- 基础 name 搜索
- 简单 token / 前缀匹配
- 搜索结果转成 `SearchEntryResultSet`

### 搜索冷启动与不完整索引策略

Phase 2 不能假设索引天然完整。

需要明确三种状态：

- 已完全索引
- 部分已索引
- 尚未建立索引

第一阶段建议的用户语义：

- 优先返回“当前已索引范围”的结果
- UI 可显示“索引构建中”或等价状态
- 不在 Phase 2 默认回退成全盘即时扫描

原因：

- 一旦搜索体验在“索引查找”和“临时全盘扫描”之间反复切换，结果和性能都很难解释

所以建议明确：

- Phase 2 搜索是“索引优先，范围可能不完整”
- 全盘补齐由后台索引构建负责

### Phase 3: 增量更新

目标：

- 不再因为目录变化而整目录重建
- 受影响记录直接在主表和视图里更新

需要：

- 稳定 `entry_id`
- watcher / USN 提供的变更信号
- 排序视图支持插入/删除/重排

### 增量更新失败回退粒度

“回退局部重建”必须写清范围。

建议默认粒度定义为：

- 受影响目录级

也就是：

- 单条变更处理失败
- Journal gap
- watcher 事件不完整
- 局部 diff 无法收敛

默认回退到：

- 受影响目录结果集重建

而不是：

- 整卷重建
- 全局全量重建

这样可以把失败成本限制在可接受范围内。

### Phase 4: 多排序视图与筛选

范围：

- 修改时间
- 大小
- 路径
- 后续可加类型/扩展名/标签

## 动态更新策略

动态更新不是“不要 watcher”，而是“收到 watcher/USN 信号后，不再整目录失效”。

推荐模型：

1. watcher / USN 提供变化信号
2. 定位受影响目录或受影响记录
3. 主表增量更新：
   - add
   - remove
   - update
   - rename
4. 排序视图局部维护
5. 任一步失败，退回局部重建

### USN 不可用与 gap 降级路径

USN 不能被当成始终可靠。

需要明确这几类情况：

- 当前环境没有 USN 能力
- 权限不足
- Journal gap / overflow
- 卷重建导致的历史断裂

建议策略：

- USN 可用时：
  - 用它提供更细的变更依据
- USN 不可用时：
  - watcher 继续作为触发器
  - 回退到目录级 diff 或目录级重建
- 检测到 gap / overflow 时：
  - 不继续信任增量链
  - 对受影响目录做一次目录级重建

这样可以保证：

- 没有 USN 也能工作
- 有 USN 时只是更准，而不是唯一依赖

## 目录级缓存与全局索引的关系

目录级持久结果集缓存不是白做的，可以作为过渡层继续存在。

建议关系：

- L0：当前请求窗口/视口缓存
- L1：会话内目录热快照
- L2：目录级持久结果集缓存
- L3：全局主索引与视图

## 分层缓存一致性规则

这是后续实现里必须先定下来的地方。

建议规则：

- L3 全局主索引与视图是长期真源
- L2 目录级持久结果集缓存是加速层，不是最终真源
- L1 会话内目录热快照是运行时缓存
- L0 当前窗口/视口缓存只服务当前 UI

一致性处理建议：

- L3 更新后：
  - 相关 L2/L1/L0 视图应标记为过期或按需重建
- L2 命中时：
  - 可以用于启动加速
  - 但不能反向覆盖一个更新代际更高的 L3
- L1/L0 永远服从上层真源或版本号判断

第一阶段可先用“版本戳/代际号”做粗同步：

- 每次目录重建或全局索引更新时递增
- 上层代际落后则直接丢弃

### L2 的退出时机

L2 不会永久消失，但会逐步降级成：

- 启动加速层
- 回退路径

建议把“L2 可降级”的判断标准写成可观测指标，例如：

- 当 L3 对常见目录首屏冷启动已经稳定低于目标延迟
- 并且目录浏览主路径已经长期不再依赖 L2

这时可以把 L2 从主路径降为加速/容灾路径

后续成熟后，可以逐步让：

- 目录浏览优先走 L3 的目录视图
- L2 退成启动加速或回退路径

## 数据结构建议总结

不要用“单一结构解决一切”。

建议组合：

- 主表：
  - `slotmap/slab + HashMap`
- 排序视图：
  - order-statistics tree
  - 第一阶段可临时 `Vec<entry_id>`
- 搜索索引：
  - 倒排/前缀索引

## 验收标准

- 目录浏览和搜索都能走统一 `ResultSet`
- `ReadRange(start, count)` 对目录和搜索都成立
- 单条变更不再默认触发整目录重建
- 搜索结果不依赖当前目录
- 多排序视图不复制整份主记录

## 性能与资源验收建议

功能验收之外，需要补一组可量化指标。

建议第一版先用这些门槛：

- `ReadRange(0, 50)` 在常见已索引结果集下保持低延迟
- 单条增量更新反映到可见结果集的延迟保持在可接受范围
- 主表与排序视图的内存占用可被持续观测

当前文档先不把具体数值写死到实现承诺里，但建议在原型阶段用基准测试把这些指标钉出来：

- 首屏范围读取延迟
- 单条 add/remove/update/rename 延迟
- 百万级记录下的内存占用

## 不在第一阶段做的内容

- 非 NTFS 全量统一索引
- 复杂全文/模糊搜索优化
- 网络路径完整支持
- 完整 Everything 级功能对齐
- 最终版增量排序维护

## 建议的下一步实现顺序

1. 定义全局索引核心类型：
   - `GlobalEntryId`
   - `GlobalEntryRecord`
   - `GlobalIndexStore`
2. 起一个只支持名称排序的全局视图原型
3. 先让 `SearchEntryResultSet` 能接入全局索引
4. 再考虑目录浏览如何逐步从目录级缓存过渡到全局视图
