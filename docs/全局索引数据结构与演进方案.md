# 全局索引数据结构与演进方案

## 目标

- 在当前“目录级持久结果集缓存”之上，演进到可跨目录复用的全局索引。
- 保持 WinUI 侧现有 `IEntryResultSet` / `ReadRange(startIndex, count)` 模型不变。
- 让目录浏览、搜索结果、后续筛选视图逐步共用同一套底层数据源。
- 为后续动态更新打基础，但第一阶段不直接追求完整 Everything 级能力。

## 当前状态

- 当前目录结果集本质上是：
  - `path + sort_mode -> Vec<MemoryDirItem>`
  - 持久化后恢复成排序好的目录级数组
- 当前优点：
  - 单目录范围读取快
  - 热目录复用明显
  - 已能跨会话命中
- 当前限制：
  - 真源和排序视图耦合
  - 不适合高频插入/删除/重命名
  - 不适合做跨目录搜索
  - 不适合多排序共享一套基础数据

## 设计原则

- 不直接把“目录级排好序数组”放大成“全局超大数组”。
- 主记录表、排序索引、搜索索引分层。
- `ResultSet` 面向 UI，索引实现细节只在 Rust 内部。
- 第一阶段优先支持：
  - 本地 NTFS
  - 名称排序
  - 基础搜索
  - 稳定随机访问
- 任何增量维护失败时，都必须允许回退到局部重建。

## 总体结构

全局索引建议拆成三层。

### 1. 主记录表

真源，不按任何排序保存。

- `entry_id`
  - 本地 NTFS 优先 `FRN/FileId`
  - 非 NTFS 或拿不到时退回内部生成 id
- `parent_id`
- `volume_id`
- `normalized_path`
- `name`
- `is_dir`
- `is_link`
- `size_bytes`
- `modified_unix_ms`
- 预留：
  - `extension`
  - `attributes`
  - `usn`

建议结构：

- `slotmap/slab + HashMap<entry_id, slot>`

原因：

- 需要稳定 id
- 需要快速按 id 找记录
- 删除后不希望让大块数组整体搬移

### 2. 排序索引

面向“按顺序取第 N 条”和“范围读取”。

每种排序一个独立有序视图，例如：

- 名称升序
- 修改时间降序
- 大小降序
- 路径升序

建议结构：

- order-statistics tree
  - 节点存 `entry_id`
  - 带子树大小

这样可以支持：

- `get nth`
- `read range(start, count)`
- 插入/删除
- 局部重排

如果第一阶段实现成本太高，也可以先用：

- `Vec<entry_id>` 作为排序视图

但这更适合作为过渡，不是最终结构。

### 3. 搜索索引

排序索引不负责搜索。

搜索索引建议单独维护：

- 前缀索引
- 倒排索引
- 或者更轻量的 token / n-gram 索引

第一阶段可先只做：

- `name` 分词/归一化后的倒排
- 基础前缀匹配

搜索结果本质上还是：

- 一组 `entry_id`
- 再套用排序视图或临时排序结果

## 与 ResultSet 的关系

UI 不直接碰全局索引内部结构。

### 目录浏览

- `DirectoryEntryResultSet`
  - 先可继续走目录级结果集缓存
  - 后续切到“全局索引中的目录视图”

### 搜索结果

- `SearchEntryResultSet`
  - 直接由搜索索引 + 排序视图产出

### UI 契约保持不变

- `TotalCount`
- `TryReadRange(startIndex, count)`

也就是说，全局索引的目标不是改 UI，而是替换 `ResultSet` 背后的数据来源。

## 分阶段演进

### Phase 1: 主表 + 名称排序视图

目标：

- 先把“目录级缓存”和“基础全局主表”接起来
- 只支持最常用的名称排序
- 优先保证目录浏览与基础搜索可共用一套底层主表

范围：

- 本地 NTFS
- `entry_id + path + name + is_dir + size + modified`
- 名称排序视图
- `ReadRange` 可直接从排序视图取 `entry_id` 后映射到主记录

### Phase 2: 搜索索引

目标：

- 在不改 UI 的前提下，把搜索结果从“目录扫描”推进到“全局索引查询”

范围：

- 基础 name 搜索
- 简单 token / 前缀匹配
- 搜索结果转成 `SearchEntryResultSet`

### Phase 3: 增量更新

目标：

- 不再因为目录变化而整目录重建
- 受影响记录直接在主表和视图里更新

需要：

- 稳定 `entry_id`
- watcher / USN 提供的变更信号
- 排序视图支持插入/删除/重排

### Phase 4: 多排序视图与筛选

范围：

- 修改时间
- 大小
- 路径
- 后续可加类型/扩展名/标签

## 动态更新策略

动态更新不是“不要 watcher”，而是“收到 watcher/USN 信号后，不再整目录失效”。

推荐模型：

1. watcher / USN 提供变化信号
2. 定位受影响目录或受影响记录
3. 主表增量更新：
   - add
   - remove
   - update
   - rename
4. 排序视图局部维护
5. 任一步失败，退回局部重建

## 目录级缓存与全局索引的关系

目录级持久结果集缓存不是白做的，可以作为过渡层继续存在。

建议关系：

- L0：当前请求窗口/视口缓存
- L1：会话内目录热快照
- L2：目录级持久结果集缓存
- L3：全局主索引与视图

后续成熟后，可以逐步让：

- 目录浏览优先走 L3 的目录视图
- L2 退成启动加速或回退路径

## 数据结构建议总结

不要用“单一结构解决一切”。

建议组合：

- 主表：
  - `slotmap/slab + HashMap`
- 排序视图：
  - order-statistics tree
  - 第一阶段可临时 `Vec<entry_id>`
- 搜索索引：
  - 倒排/前缀索引

## 验收标准

- 目录浏览和搜索都能走统一 `ResultSet`
- `ReadRange(start, count)` 对目录和搜索都成立
- 单条变更不再默认触发整目录重建
- 搜索结果不依赖当前目录
- 多排序视图不复制整份主记录

## 不在第一阶段做的内容

- 非 NTFS 全量统一索引
- 复杂全文/模糊搜索优化
- 网络路径完整支持
- 完整 Everything 级功能对齐
- 最终版增量排序维护

## 建议的下一步实现顺序

1. 定义全局索引核心类型：
   - `GlobalEntryId`
   - `GlobalEntryRecord`
   - `GlobalIndexStore`
2. 起一个只支持名称排序的全局视图原型
3. 先让 `SearchEntryResultSet` 能接入全局索引
4. 再考虑目录浏览如何逐步从目录级缓存过渡到全局视图
