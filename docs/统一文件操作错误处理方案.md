# 统一文件操作错误处理方案

## 背景

当前文件管理命令已经覆盖了创建、重命名、删除、复制/剪切/粘贴、创建快捷方式、ZIP 压缩等操作，但错误处理仍然偏分散：

- 有些命令直接显示原始异常消息。
- 有些命令只返回“失败”，缺少明确原因。
- 不同命令的状态文案风格不一致。
- 后续如果继续增加 ZIP 解压等能力，错误面会进一步扩大。

因此需要先收口一层统一的文件操作错误模型，再继续扩展解压菜单等功能。

### 本轮补充

- 新增 `Canceled` 错误分类。
- 对 `OperationCanceledException` 和 Win32 `1223 (ERROR_CANCELLED)` 做了统一映射。
- 删除链路在系统错误对话框中被用户取消后，当前会显示“已取消”语义，而不再落成“未知错误”。

### 当前保留的系统行为

- 删除仍保持系统资源管理器语义：
  - 继续走回收站删除
  - 继续允许系统自己弹文件占用/错误对话框
- 因为底层仍是系统删除链路，不同外部进程的文件共享模式可能导致：
  - 有时直接删除成功
  - 有时弹出系统“文件正在使用”对话框
- 这是刻意保留的系统行为，不在当前阶段改成完全应用内自定义删除流程。

## 目标

- 统一文件操作错误分类。
- 统一用户可见状态文案。
- 保留底层异常细节，但不直接把原始异常暴露给普通用户。
- 让现有命令和后续命令都走同一套结果结构。

## 覆盖范围

第一阶段覆盖以下命令：

- `copy`
- `cut`
- `paste`
- `delete`
- `rename`
- `new-file`
- `new-folder`
- `create-shortcut`
- `compress-zip`
- 后续 `extract-*`

## 建议分层

### 结果层

定义统一结果对象，例如：

- `FileOperationResult.Success(...)`
- `FileOperationResult.Failure(FileOperationError error, string? detail = null)`

结果对象只表达：

- 是否成功
- 用户语义上的错误类型
- 可选的技术细节
- 可选的附带信息
  - 目标路径
  - 受影响文件名
  - 是否部分成功

### 错误层

当前错误集以有限集合为主，不追求一次覆盖所有异常类型：

- `NotFound`
- `AccessDenied`
- `AlreadyExists`
- `InvalidName`
- `PathTooLong`
- `InUse`
- `DiskFull`
- `Canceled`
- `ArchiveInvalid`
- `ArchiveEntryUnsafe`
- `PartialFailure`
- `Unknown`

说明：

- `ArchiveInvalid`、`ArchiveEntryUnsafe` 现在先放进去，是为了下一阶段 ZIP 解压时直接复用。
- `PartialFailure` 用于批量粘贴、批量解压这类可能半成功半失败的命令。

### 映射层

在 `ExplorerService` / `FileManagementCoordinator` 与 `MainWindow` 之间增加统一映射：

- 系统异常 / IO 异常 / COM 异常 / Shell 失败
- 映射成统一的 `FileOperationError`

这样 `MainWindow` 不再自己判断大量异常类型，只负责：

- 调命令
- 读取结果
- 显示状态
- 必要时执行后续 UI 动作

## 用户文案原则

- 优先显示稳定的业务文案，不直接显示原始英文异常。
- 同一类错误在不同命令下保持一致表达。
- 有必要时附带短技术细节，但不把原始堆栈或低层异常直接塞进状态栏。

示例：

- `无法删除：文件正在被其他程序使用`
- `无法重命名：名称无效`
- `无法粘贴：目标位置已存在同名项目`
- `无法压缩：当前文件夹不可用`
- `无法解压：压缩包已损坏`

## 与现有代码的结合方式

当前实现仍保持现有 UI 结构，主要做命令返回值和用户文案收口：

- `ExplorerService`
  - 逐步把直接抛异常的文件操作入口，升级为返回统一结果，或由协调层统一包装。

- `FileManagementCoordinator`
  - 优先作为第一层统一错误收口点。
  - 适合承接 create / rename / delete / paste / shortcut / extract 等跨 UI 命令。

- `MainWindow`
  - 尽量不再自己拼异常消息。
  - 只消费统一结果，并更新状态栏、刷新列表、恢复选中/视口。

## 后续建议

- 当前底座已经能支撑 ZIP 解压第一阶段。
- 后续更值得继续收口的是：
  - 是否为关键失败补 `InfoBar` 或更强提示，而不只停留在状态栏
  - 是否把更多系统集成型命令结果也统一收敛到同一个结果模型，而不只是用户文案映射
  - 是否补批量操作的部分成功结果展示

## 非目标

当前阶段先不做：

- 独立错误弹窗系统
- 全量操作日志 UI
- 所有历史命令一次性全部重构
- 7z/tar 解压支持

当前目标是先把最常用文件操作的失败语义收口一致。
