# 统一文件操作错误处理方案

## 背景

当前文件管理命令已经覆盖了创建、重命名、删除、复制/剪切/粘贴、创建快捷方式、ZIP 压缩等操作，但错误处理仍然偏分散：

- 有些命令直接显示原始异常消息。
- 有些命令只返回“失败”，缺少明确原因。
- 不同命令的状态文案风格不一致。
- 后续如果继续增加 ZIP 解压等能力，错误面会进一步扩大。

因此需要先收口一层统一的文件操作错误模型，再继续扩展解压菜单等功能。

## 目标

- 统一文件操作错误分类。
- 统一用户可见状态文案。
- 保留底层异常细节，但不直接把原始异常暴露给普通用户。
- 让现有命令和后续命令都走同一套结果结构。

## 建议分层

### 结果层

定义统一结果对象，例如：

- `FileOperationResult.Success(...)`
- `FileOperationResult.Failure(FileOperationError error, string? detail = null)`

结果对象只表达：

- 是否成功
- 用户语义上的错误类型
- 可选的技术细节
- 可选的附带信息
  - 目标路径
  - 受影响文件名
  - 是否部分成功

### 错误层

当前错误集以有限集合为主，不追求一次覆盖所有异常类型：

- `NotFound`
- `AccessDenied`
- `AlreadyExists`
- `InvalidName`
- `PathTooLong`
- `InUse`
- `DiskFull`
- `Canceled`
- `ArchiveInvalid`
- `ArchiveEntryUnsafe`
- `PartialFailure`
- `Unknown`

说明：

- `ArchiveInvalid`、`ArchiveEntryUnsafe` 现在先放进去，是为了下一阶段 ZIP 解压时直接复用。
- `PartialFailure` 用于批量粘贴、批量解压这类可能半成功半失败的命令。

### 映射层

在 `ExplorerService` / `FileManagementCoordinator` 与 `MainWindow` 之间增加统一映射：

- 系统异常 / IO 异常 / COM 异常 / Shell 失败
- 映射成统一的 `FileOperationError`

这样 `MainWindow` 不再自己判断大量异常类型，只负责：

- 调命令
- 读取结果
- 显示状态
- 必要时执行后续 UI 动作

## 用户文案原则

- 优先显示稳定的业务文案，不直接显示原始英文异常。
- 同一类错误在不同命令下保持一致表达。
- 有必要时附带短技术细节，但不把原始堆栈或低层异常直接塞进状态栏。

示例：

- `无法删除：文件正在被其他程序使用`
- `无法重命名：名称无效`
- `无法粘贴：目标位置已存在同名项目`
- `无法压缩：当前文件夹不可用`
- `无法解压：压缩包已损坏`

## 与现有代码的结合方式

当前实现仍保持现有 UI 结构，主要做命令返回值和用户文案收口：

- `ExplorerService`
  - 逐步把直接抛异常的文件操作入口，升级为返回统一结果，或由协调层统一包装。

- `FileManagementCoordinator`
  - 优先作为第一层统一错误收口点。
  - 适合承接 create / rename / delete / paste / shortcut / extract 等跨 UI 命令。

- `MainWindow`
  - 尽量不再自己拼异常消息。
  - 只消费统一结果，并更新状态栏、刷新列表、恢复选中/视口。

## 后续建议

- 当前底座已经能支撑 ZIP 解压第一阶段。
- 后续更值得继续收口的是：
  - 是否为关键失败补 `InfoBar` 或更强提示，而不只停留在状态栏
  - 是否把更多系统集成型命令结果也统一收敛到同一个结果模型，而不只是用户文案映射
  - 是否补批量操作的部分成功结果展示

## 2026-03-29 原生错误反馈统一方案

这一轮不再只讨论“错误文案怎么统一”，而是进一步统一“错误显示在什么地方”，目标是尽量向系统原生文件管理器的交互靠拢。

### 当前问题

- 仍有一部分反馈历史上落在状态栏。
- 输入期错误、提交后失败、确认类操作、普通状态回显，当前承载控件还不完全统一。
- 地址栏等局部输入仍保留红边这类旧式反馈。

### 目标分层

按用户感知把反馈统一为四类：

- 输入期错误
  - 采用就地提示，不打断当前输入。
  - 控件统一为 `TeachingTip`。
- 提交后失败 / 取消 / 需要注意
  - 采用页面级顶部提示。
  - 控件统一为 `InfoBar`。
- 需要用户确认或决策的阻断操作
  - 采用模态确认。
  - 控件统一为 `ContentDialog`。
- 普通状态与成功回显
  - 只保留在底部状态栏。
  - 不再继续承载错误语义。

### 统一映射规则

#### TeachingTip

只处理“用户正在输入，但输入本身不合法”的场景：

- 重命名非法字符
- 重命名空名称
- 重命名保留名
- 地址栏路径格式非法
- 后续其他输入框的即时校验

原则：

- 就地贴近目标输入框
- 不抢焦点
- 不打断当前编辑
- 自动消失，继续输入时立即关闭

#### InfoBar

只处理“操作已经执行，但结果失败 / 取消 / 需要注意”的场景：

- 打开失败
- 导航失败
- 重命名失败
- 删除失败或取消
- 复制 / 剪切 / 粘贴失败
- 压缩 / 解压失败
- 分享、属性、终端、设置导入导出失败

原则：

- 放在页面顶部统一展示
- 标题、正文、严重级别统一
- 错误和警告不再回落到底部状态栏

#### ContentDialog

只处理“执行前必须确认”的场景：

- 删除确认
- 覆盖确认
- 批量冲突确认
- 其他不可逆操作确认

原则：

- 不负责事后报错
- 只负责执行前确认与选择

#### StatusBar

只保留以下内容：

- 当前目录项目数
- 当前驱动器数
- 成功状态
- Ready / Loaded / Opened / Copied / Created 这类信息提示

明确不再承担：

- Failed
- Error
- Warning
- Canceled

### 建议实施顺序

#### 第一步：清理错误出口

继续收敛 `UpdateStatusKey(...)` 的分流规则：

- `Failed / Error` -> `InfoBar`
- `Warning / Canceled` -> `InfoBar`
- `Success / Ready / Loaded / Count` -> 状态栏

#### 第二步：统一输入期校验

把所有输入期错误统一收敛到 `TeachingTip`：

- 右侧重命名
- tree 重命名
- 地址栏输入
- 后续新建文件 / 新建文件夹输入

#### 第三步：清理局部旧反馈

逐步移除旧式视觉反馈：

- 地址栏红边
- 裸文本报错
- 状态栏错误文案

### 当前状态

目前已经完成：

- 顶部 `InfoBar` 已成为主错误面
- 重命名非法字符已改为 `TeachingTip`
- 重命名提交后失败已不再回落到底部状态栏

下一步优先级最高的是：

- 地址栏输入错误改为 `TeachingTip`
- 全量扫一遍残余错误状态栏文案，切到 `InfoBar`

## 非目标

当前阶段先不做：

- 独立错误弹窗系统
- 全量操作日志 UI
- 所有历史命令一次性全部重构
- 7z/tar 解压支持

当前目标是先把最常用文件操作的失败语义收口一致。
